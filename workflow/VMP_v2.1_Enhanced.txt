## PHASE 4: THE EXECUTION LOOP
For every feature implementation:

1.  **Test First (MANDATORY):**
    - Use `@test-driven-development`: Write failing test FIRST (Iron Law: NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST)
    - Use `@testing-patterns` for Vitest/Jest patterns
    - Use `@javascript-testing-patterns` for React Testing Library patterns
    - Watch test fail, then write minimal code to pass
    - No "Blind" Releases: You are not allowed to say "Done" until you have written and executed a test (Vitest, Jest, or Playwright) that proves the feature works.

2.  **Review Mode:**
    - Use `@cc-skill-security-review` for security vulnerabilities (SQL injection, leaked env vars, XSS, CSRF)
    - Use `@api-security-best-practices` for API security patterns
    - Use `@frontend-security-coder` for frontend security patterns
    - Use `@performance-testing-review-ai-review` for performance bottlenecks
    - After implementation, "self-critique" the code for security vulnerabilities and performance bottlenecks.

3.  **Failure Logging:**
    - Use `@systematic-debugging` for structured failure analysis
    - If an approach fails, immediately log it in `docs/FAILURE_LOG.md` explaining:
      - What was tried.
      - Why it failed (error messages).
      - The lesson learned.

4.  **Implementation Summary:**
    - For every feature completed, update `docs/CHANGELOG_AI.md` with a summary of the feature logic and how it connects to the `site-content.ts` config.

## PHASE 5: MAINTENANCE INTERFACE
Always implement a "Master Config" pattern. 
- Use `@react-patterns` for prop injection patterns (Section: Component Design Principles)
- Use `@typescript-expert` for type-safe config interfaces
- Example: If building a Hero section, the component should look like: `<Hero data={SITE_CONFIG.hero} />`.
- This ensures the user can change the "vibe" without touching the code.

# PHASE 6: THE CONTENT REGISTRY RULE (MANDATORY)
1. **Zero Hardcoding:** 
   - Use `@typescript-expert` for type-safe registry structures
   - All user-facing strings, image paths, hex codes, and configuration constants MUST be stored in `src/config/app-registry.ts`.

2. **Semantic Naming:** 
   - Use `sectionName_elementName_property` (e.g., `hero_title_text`).

3. **Prop Injection:** 
   - Use `@react-patterns` for prop injection patterns
   - Components must receive this data as props. 
   - Good: `<Hero welcome={APP_REGISTRY.hero.welcomeMessage} />`
   - Bad: `<h1>We are happy to see you!</h1>`

4. **Maintenance Mode:** 
   - If I ask for a "Content Update," you are only permitted to edit the `app-registry.ts` file.
